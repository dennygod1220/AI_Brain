以下是這部影片的核心技術脈絡與精華總結：

### 1. 破除主流架構迷思：「Plan and Execute」未必永遠適用

- **初始痛點**：作者最初依照市面上的標準教學，搭建了「主 Agent (負責規劃/調度) + 子 Subagent (負責設計/寫程式)」的層級架構。但在實務中發現，面對小任務時，節點間的通訊成本極高（流程太重），且主 Agent 經常無法守住邊界，會越權自己去修改程式碼。
    
- **第一次重構 (扁平化)**：作者放棄了子 Agent，改用近期熱門的 **「Skill」概念（動態提示詞注入）**。將原本交給子任務的指引，直接作為上下文注入單一 Agent 中。結果發現系統變得更輕量、順暢，Token 消耗也減少了。這證明了盲目追求多層級架構有時只是一種「架構強迫症」。
    

### 2. 長任務 (Long-running Tasks) 帶來的新災難

當要求系統連續生成大量內容（如上百段程式碼）時，模型出現了三大崩壞現象：

1. **偷懶與合併**：模型開始將步驟合併，急於「趕工」。
    
2. **規則失效**：原先設定好的風格和流程約束，越到後期越被當成空氣。
    
3. **高度同質化**：AI 失去創意，開始不斷複製貼上自己前面寫過的程式碼架構。
    

### 3. 核心洞察：底層原因是「注意力機制 (Attention)」與「上下文焦慮」

- **問題本質**：大模型對上下文的注意力是不平均的。它通常只高度關注最前面的 `System Prompt` 以及「距離當前輸出最近」的內容。中間的規則一旦被大量生成的內容淹沒，模型就會「視而不見」。
    
- **解法 (Restatement 重申機制)**：作者引入了 To-Do List，但其真正發揮作用的關鍵不在於「記錄」，而在於 **「不斷重申 (Restatement)」**。
    
    - **靜態規則**（不變的規範）放在 `System Prompt`。
        
    - **動態規則/當前進度**（如 Skill 或具體步驟）必須**不斷被推到最新生成內容的尾端**。利用這個機制強佔模型的「注意力高地」，配合 KV Cache（鍵值緩存）的機制，強制模型在每一步都回顧最重要的約束。
        

### 4. 再次反轉：為了「上下文隔離」喚回 Subagent

- 為了解決前面提到的「輸出同質化」問題（因為模型看到了自己前面寫的所有程式碼，產生了思維定勢），作者進行了第二次架構調整——**把 Subagent 加回來**。
    
- **關鍵差異**：這次加回 Subagent **不是**為了做任務分發，而是為了創造一個 **「乾淨的上下文視窗 (Context Window)」**。透過隔離不必要的歷史資訊，強迫 AI 在局部任務中獨立思考，恢復其生成的多樣性與創意。

### 為什麼光靠 `soul.md` 重申還不夠？（本地小模型的致命傷）

你目前的主要場景是「爬網站、分析內容、總結」。這在架構上會面臨一個業界稱為 **「上下文腐敗 (Context Rot)」** 的現象。 當你把爬蟲抓回來的幾千字網頁內容塞進去時，本地小模型的注意力（Attention）會瞬間被這些海量噪聲給沖散。這時候，就算你在最後加上一句「請遵守 `soul.md` 的規則」，模型的大腦也已經超載（Context Window 擠爆），它會開始出現「幻覺（Hallucination）」，甚至忘記自己是誰。

要徹底解決這個問題，我們不能只靠「重申」，必須導入最新的 **Agent = Model + Harness（外圍框架）** 的思維。你可以採取以下三個具體的優化步驟：

### 具體優化方案：3 步拯救你的 Hermes Agent

#### 1. 隔離污染源：用 Sub-agent 處理網頁（就是影片中把 Subagent 加回來的真正用意）

不要讓你的「主 Agent」直接去讀取爬蟲抓回來的原始網頁。 在 Hermes 等現代框架中，你應該使用「委派（Delegation）」模式。

- **作法**：當需要分析網站時，主 Agent 呼叫一個臨時的 Sub-agent。這個 Sub-agent 的唯一任務就是：「讀取這篇網頁，並根據我的要求，萃取出 3-5 點關鍵資訊，然後把其他廢話丟掉。」
    
- **好處**：主 Agent 的上下文永遠保持乾淨，它只會看到經過濃縮的「高價值資訊」，小模型的智商就能持續在線。
    

#### 2. 卸載記憶體：把大腦外包給「檔案系統 (Filesystem)」

你提到 Agent 在連續任務後會開始搞事情，因為歷史對話太長了。

- **作法**：不要把所有總結好的資料一直留在聊天紀錄（Context）裡。給 Agent 一個寫入硬碟的工具（例如寫入 `research_notes.md`）。當 Sub-agent 總結完資料後，直接存進本地檔案。
    
- **好處**：你的 Agent 每次思考時，上下文裡只有「當前的任務目標」，如果它需要過去的資料，再讓它去讀取那個 `.md` 檔。這大幅減輕了小模型的 Token 壓力。
    

#### 3. 正確的 Restatement（你的 `soul.md` 該怎麼用）

回到你的 `soul.md` 想法，這是對的，但要「切分」與「動態注入」。

- **靜態規則（絕對不能變的，如：輸出格式、人設）**：放在一開始的 System Prompt 裡。
    
- **動態規則（如：當前的 Skill、現在要進行的步驟）**：這才是 `soul.md` 該發揮作用的地方。但不要讓 Agent 自己去「讀」它，而是**透過你的 Python 腳本或框架（Harness），在每次發送給 LLM 的「最新一條 User Prompt」的尾端，自動附加上去**。
    
- **範例提示詞**：「_（用戶的指令或爬蟲結果）... \n\n【系統提示：你目前位於研究階段的第 2 步，請嚴格遵守以下 Skill 規則：...】_」
    

### 總結

你目前的思路已經走在正確的道路上。優化的關鍵在於：**不要跟小模型的記憶體硬碰硬。** 用 Sub-agent 當作過濾器來隔離長篇網頁，把收集到的資料存到本地 `.md` 檔（而不是留在對話裡），最後在每次對話的最尾端，用程式碼動態注入當下最需要的 Skill 規則。

這樣一來，你的本地 Hermes Agent 就能用極低的成本，跑出媲美雲端大模型的穩定長任務表現。如果你在具體實作 Sub-agent 的串接或提示詞的撰寫上有遇到困難，可以隨時告訴我，我們再來推演具體的代碼邏輯！

### 只有單一provider時，分派subagent上下文也是會分開的嗎?
>我問: 
>那假設我就只有一個本地llm(假設我用的是llamacpp -np只有1)，並且也沒有足夠的資源可以讓他併行運算，當主agent分派任務給子agent時，其實還是同一個llm，那上下文是會分開的嗎?

這是一個非常直指核心的架構問題！身為前瞻數據研究員，我可以給你一個明確且絕對肯定的答案：

**是的，即使你底層只有一個 `llama.cpp` 實例，而且不能並行運算（也就是每次只能處理一個請求），「主 Agent」與「子 Agent」的上下文（Context）依然是完全分開的！**

為了讓你完全掌握這個概念，我們必須破解一個關於 LLM 運作原理的常見直覺迷思。以下為你拆解這背後的底層邏輯：

### 1. 核心觀念：LLM 是「無狀態的 (Stateless)」

很多人會把 LLM 想像成一個「持續在聽你說話的人」，但實際上，**LLM 是一個「沒有短期記憶的函數」**。

每一次你呼叫 `llama.cpp` 生成文字時，對模型來說都是一個「全新的宇宙」。它之所以看起來有記憶，是因為你的外圍程式（如 Python 腳本）在每次發送請求時，都把**整個歷史對話紀錄重新傳給了它**。

因此，主 Agent 和子 Agent 是不是同一個模型，根本不重要。重要的是**「你這次傳給它的 Prompt 長什麼樣子」**。

### 2. 情境比喻：廚房裡唯一的大廚 (np=1)

假設你的 `llama.cpp` 是一個技術高超但沒有記憶力的廚師。你只有一個廚師，而且他一次只能做一件事。

- **主 Agent 發問**：你遞給大廚一張單子（Prompt A：「你是主 Agent，請決定下一步...」）。大廚看完單子，給出答案，然後**立刻忘記剛剛發生過的所有事**。
    
- **子 Agent 發問**：接著，你遞給他另一張全新的單子（Prompt B：「你是網頁萃取器，請分析這段 HTML...」）。大廚拿過這張新單子，照著做，然後給出答案。
    

在這個過程中，大廚（模型）是同一個，但他處理的單子（Context）是完全隔離的。因為是順序執行（不能並行），所以他就是先做完主 Agent 的工作，再去做子 Agent 的工作，互不干涉。

### 3. 在程式碼層面，這具體長什麼樣子？

在你的 Python 腳本裡，你只需要維護**兩個不同的 List（陣列）** 來存放對話紀錄：

Python

```
# 這是主 Agent 的大腦（記憶體）
main_agent_context = [
    {"role": "system", "content": "你是規劃全局的主 Agent..."},
    {"role": "user", "content": "我們現在進行到第 2 步..."}
]

# 這是子 Agent 的大腦（用完即丟）
sub_agent_context = [
    {"role": "system", "content": "你是網頁分析子 Agent，請嚴格萃取資訊。"},
    {"role": "user", "content": "<這裡塞入超長的網頁 HTML>"}
]

# 輪到主 Agent 思考：把 main_agent_context 丟給 llama.cpp
response_main = llm.generate(main_agent_context) 

# 輪到子 Agent 工作：把 sub_agent_context 丟給同一個 llama.cpp
response_sub = llm.generate(sub_agent_context) 
```

如你所見，因為變數是分開的，只要你不把 `sub_agent_context` 裡的長篇網頁 HTML 塞進 `main_agent_context` 裡，主 Agent 的腦袋就永遠是乾淨的。

### 4. 唯一的代價：時間與 KV Cache 重算

既然上下文不會互相污染，那用單一個本地小模型這樣切換，有沒有缺點？

有，代價在於**運算時間（運算資源）**。 為了加速，現代推論引擎（包含 `llama.cpp`）會使用一種叫做 `KV Cache` 的技術來快取剛才算過的上下文。 當你頻繁在「主 Agent Prompt」和「子 Agent Prompt」之間切換時，模型發現前文完全變了，它就必須清空緩存，重新計算整段 Prompt（這稱為 Prefill 階段）。

- **結果就是**：子 Agent 開始吐出第一個字的時間（Time to First Token, TTFT）會稍微變長一點，因為它要重新消化那篇長網頁。
    

### 結論

你完全可以、也強烈建議在你目前的本地環境中實作「主/子 Agent 職責分離」。

只要在你的腳本裡，**把傳遞給模型的 Prompt 切分成兩個獨立的變數**，你就能完美解決上下文污染的問題。就算只有單卡、單執行緒（np=1），你的 Agent 系統依然能保持極高的邏輯清晰度。
