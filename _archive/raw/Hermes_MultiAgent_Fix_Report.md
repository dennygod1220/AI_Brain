# Hermes 多 Agent (Discord) 通訊與協作修復總結報告

這份報告詳細記錄了解決主 Agent (蝦蝦) 與本地 Agent (小低能) 在 Discord 環境下跨 Agent 溝通障礙的完整分析與處理過程。

---

## 階段一：底層通訊架構與權限打通

### 1. Bot 訊息靜音 (白名單過濾)
- **現象**：主 Agent 發送標記消息給小低能，但小低能完全無反應該，Log 亦無入站紀錄。
- **分析**：Hermes 的 Discord Adapter 預設不信任並過濾掉其他 Bot 的訊息，防止發生無限對話迴圈。
- **處理方式**：
  - 更新兩個 Agent 的 `.env` 檔案，於 `DISCORD_ALLOWED_USERS` 加註雙方 ID 與使用者 ID。
  - 將過濾規則設定為 `DISCORD_ALLOW_BOTS=mentions`，確保只有在被標記時才接受其他 Bot 的訊息。

### 2. 環境衝突導致的啟動崩潰
- **現象**：主 Agent Systemd 服務重啟時出現 `ModuleNotFoundError: No module named 'yaml'`，進入 Crash Loop。
- **分析**：UV 等套件管理器創建的 Python 虛擬環境 (venv) 未被 Systemd 正確識別，系統回退使用本機的 Python 解析器而遺失依賴。
- **處理方式**：設定 Override 配置，將 `ExecStart` 的執行路徑明確綁定為 `/root/.hermes/hermes-agent/venv/bin/python`。

---

## 階段二：標記偵測與討論串邀請補丁

### 3. 身分組標記解析失敗 (Role Mentions)
- **現象**：使用者習慣標記身分組（如 `@小低能`），但 Agent 卻完全無視。
- **分析**：代碼中的 `on_message` 僅過濾 `message.mentions` (使用者標記)，忽略了 Discord 的 `message.role_mentions`。
- **處理方式**：修改 `gateway/platforms/discord.py`，增加對身分組標記 ID 的比對。

### 4. 討論串成員資格盲區
- **現象**：在討論串中標記小低能，小低能依舊收不到訊息。
- **分析**：在 Discord 機制中，如果你不拉 Bot 進入私有討論串，它無法獲取該頻道的對話事件。僅依靠標記身分組不會觸發「自動加入討論串」機制。
- **處理方式**：在 Hermes 的 `_auto_create_thread` 建構流程中，添加自動邀請邏輯：一但建立討論串，主動檢查訊息內容並執行隱藏的 `thread.add_user()`。

## 階段三：解決「雙機搶話」與「視覺屏障」

### 5. 多 Agent 討論串搶話 (Crosstalk)
- **現象**：一旦雙方被拉入同一個討論串，只要使用者發送一句「沒有標記任何人」的話，兩個 Agent 的大腦就會同時運作並給出互不相干的回覆，讓對話難以推進。
- **分析**：Agent 一旦具備討論串成員身分，就會監聽並嘗試回答每一句發言。
- **處理方式**：在 `discord.py` 中寫入**「嚴格沉默原則」**。只要訊息中帶有「其他 Bot 或別人的身分組標記」，且「沒有標記自己」，當前 Agent 就會強制在封包列隊級別將該訊息丟棄（Return）。這就像一個會議室的「發言權麥克風」，拿對了標籤才能說話。

### 6. 小模型幻覺與「看不到對方說話」的錯覺
- **現象**：使用者要求小低能把蝦蝦拉進討論串、或者把產出成果秀給蝦蝦看，結果小低能說「我沒有權限」，或者標記了 `@小低能` 自己。蝦蝦的回應則是「我沒看到小低能說了甚麼」。
- **分析**：
  - **視線受阻是機制，不是 Bug**：因為上述的「嚴格沉默原則」，既然小低能標記錯了對象，蝦蝦理所當然就聽不見也看不見。
  - **小模型幻覺**：這是 LLM 角色扮演的問題，小低能以為自己是普通的權限受限機器人，反而給了使用者「手動標記教學」，而非直接吐出對方 ID。
- **處理方式**：
  - 重寫小低能的 `SOUL.md`，使用極度強制、具體的正向指令。
  - 向其明確寫入蝦蝦的開發者 ID (`<@1491405019159724082>`)。
  - 告知其「拉人不需要權限，直接打出該 ID 即可」，打破它的系統限制幻覺。

## 未來協作最佳實踐
目前的架構已經達成最佳的 Discord 資源管控機制。未來的操作邏輯如下：
1. **單點指揮**：直接 `@蝦蝦` 或 `@小低能` 交付獨立任務，未被點名的一方將完全待命以節省資源。
2. **傳遞工作流**：當需要將階段任務轉交時，指令明確：「`@小低能`，請將該成果展示給 `@蝦蝦` 繼續處理。」即可維持上下文傳遞不中斷。
